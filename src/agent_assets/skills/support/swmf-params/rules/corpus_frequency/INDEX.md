# corpus_frequency/

**These files are statistical hints, not requirements.** The agent must not
treat any value in `*_typical.yaml` as authoritative.

## What's in here

Per-archetype "value envelope" files auto-generated by
`scripts/mine_param_corpus.py` from the SWMFSOLAR / SWMF Test* shipped PARAM
corpus. Each file records, for an archetype:

- The distribution of values seen for each `#COMMAND.parameter` across the
  corpus (typical range, mode, count, source files).

## What's NOT in here

The old `*_required.yaml` files (corpus-intersection "required" lists) were
deleted in the Option 2 refactor. They were the file class that hid `#CURLB0`
from the Liu et al. 2026 replication — when the SC-only intersection covered
only 2 corpus files, mandatory commands fell out of the list whenever an
archetype's mined slice was statistically small. Required-command floors now
live in `required_floors/`, populated only from PARAM.XML `<if>` conditionals
or Fortran source — never from corpus frequency.

## When the agent uses these

In **warn-tier** rule-checker rules only. Specifically:

- The `physical_constraints.yaml` rule-checker can declare a `severity=warn`
  rule that compares an emitted value against the corpus envelope (e.g.
  "`PoyntingFluxPerBSi=0.476e6` is 3σ below the AWSoM SC-only corpus median
  0.95e6 — confirm this is paper-supplied").
- The agent itself **does not** read these files as authority. They are not
  part of the swmf-replicate authoring ladder.

## Refreshing

```bash
cd swmf-mcp-prototype
python scripts/mine_param_corpus.py
```

This regenerates every `*_typical.yaml` in place. Do not hand-edit.
